# 01 · IU Task Card Template
## 内在宇宙任务卡模板（IU-TC）v0.1

**作用：**  
把任何产品、视觉、交互、状态、数据或 Agent 协作需求，压缩为一个可授权、可验证、可失败处理的最小开发单元。

**原则：**  
一个任务卡只解决一个“用户可观察结果”。  
任务卡不是需求说明书，不是技术方案，不是产品蓝图。  
它是执行 Agent 和验证 Agent 之间唯一有效的工作合同。

---

# 1. 使用规则

## 1.1 任务卡必须满足

```text
目标唯一；
范围有限；
授权明确；
约束明确；
检验可复现；
失败可分类；
停止点明确。
```

若任务卡无法让独立验证 Agent 在不询问执行者的情况下判断 Pass / Fail，则任务卡不合格。

---

## 1.2 一个任务只允许一个主变化

正确示例：

```text
“用户完成一次镜像表达后，事件被保存到 localStorage，刷新后仍存在。”
```

错误示例：

```text
“实现完整的镜像系统、候选对象、命名机制和世界持久化。”
```

任务应按用户可观察闭环拆分，而不是按开发者习惯拆分。

---

## 1.3 禁止隐含目标

任务卡中没有写出的能力，默认不允许执行 Agent 自行补全。

例如：

```text
未写“可新增依赖” → 不允许安装依赖；
未写“可创建文件” → 不允许创建文件；
未写“可修改视觉” → 不允许修改视觉；
未写“可接后端” → 不允许接后端；
未写“可重构” → 不允许重构。
```

# 3. 任务类型分类

任务类型只用于选择不同检验重点，不代表任务可以扩大。

## T1 · 视觉状态任务

适用：

```text
对象显影；
粒子变化；
微光出现；
空间雾化；
镜像薄膜；
对象关系共振。
```

重点检验：

```text
视觉变化是否可见；
变化是否由明确状态驱动；
是否符合视觉锚点；
是否像显影而不是奖励；
是否破坏宏观氛围。
```

---

## T2 · 交互状态任务

适用：

```text
靠近对象；
退出凝视；
输入表达；
沉入对象；
回到宏观；
候选对象被靠近。
```

重点检验：

```text
状态是否可达；
状态是否可退出；
是否有死路；
Escape / 返回是否安全；
回归路径是否完整。
```

---

## T3 · 数据与持久化任务

适用：

```text
保存心理事件；
保存显影等级；
保存回访次数；
保存候选对象状态；
保存用户暂时命名。
```

重点检验：

```text
数据是否写入；
刷新后是否恢复；
是否存在重复写入；
是否可解释；
是否不以聊天列表暴露历史。
```

---

## T4 · 语言行为任务

适用：

```text
镜像回应；
命名邀请；
拒绝命名回应；
离开回应；
候选对象提示。
```

重点检验：

```text
不诊断；
不归因；
不替用户命名；
不承诺疗效；
不命令用户；
保留开放性；
回到用户原话的动作、词或张力。
```

---

## T5 · 规则引擎任务

适用：

```text
心理事件 → 显影信号；
回访 → candidateSignalCount；
重复主题 → 轮廓形成；
对象关系 → 共振状态。
```

重点检验：

```text
规则可解释；
输入与输出可追溯；
无隐藏随机数；
不把语言直接压成诊断标签；
阈值可调试；
视觉变化与规则一致。
```

---

## T6 · 平台与执行纪律任务

适用：

```text
限制 Agent 文件范围；
建立本地验证流程；
加入截图输出；
创建工作树规则；
接入监督或验证 Agent。
```

重点检验：

```text
是否减少执行偏差；
是否减少未授权工具调用；
是否改善任务完成率；
是否引入额外复杂度；
是否可通过行为 benchmark 验证。
```

# 4. 首批 MVP 任务卡建议序列

以下不是执行命令，而是推荐的任务拆分顺序。

```text
IU-MVP-001
定义 localStorage 世界状态骨架。

IU-MVP-002
一次完整镜像表达后，保存 PsychologicalEvent。

IU-MVP-003
刷新后恢复琥珀显影与微光。

IU-MVP-004
重新靠近琥珀时，记录回访并沉积回访信号。

IU-MVP-005
重复事件达到阈值后，微光进入模糊轮廓阶段。

IU-MVP-006
候选结构进入可被靠近状态。

IU-MVP-007
候选结构支持“暂时命名 / 以后再说 / 先离开”。

IU-MVP-008
用户命名在刷新后仍存在，且可被修改或移除。

IU-MVP-009
完成 MVP Benchmark A–E 的全流程验证。

IU-MVP-010
进行一次失败账本复盘与规则更新评估。
```

这些任务必须逐张执行、逐张验证。  
不得把 IU-MVP-001 至 IU-MVP-008 合并成“实现 MVP”。

# 5. 示例任务卡：心理事件本地存储

```text
任务编号：IU-MVP-002
任务标题：一次镜像表达后保存心理事件
所属阶段：MVP · Persistence Layer
优先级：P0
任务类型：T3 数据与持久化
状态：Draft

────────────────────
一、目标
────────────────────

让用户完成一次“输入 → 镜像回应 → 让它沉入其中”后，
系统在浏览器本地保存一条非聊天化的 PsychologicalEvent。

────────────────────
二、完成定义
────────────────────

用户完成镜像表达并点击“让它沉入其中”后：

- 系统保存一条事件；
- 事件包含用户原话、镜像回应、当前对象、显影等级和时间；
- 用户界面不出现聊天记录列表；
- 开发期调试区能显示 eventCount 增加；
- 刷新后 eventCount 仍存在。

────────────────────
三、范围
────────────────────

包含：
- localStorage 写入；
- 心理事件对象创建；
- eventCount 调试标识；
- 既有镜像闭环的事件落点。

不包含：
- 候选对象；
- 多对象事件；
- 云端同步；
- 真实 AI；
- 可视化历史；
- 命名功能。

────────────────────
四、允许
────────────────────

可修改文件：
- inner-universe-prototype.html

可新增文件：
- 无

可使用工具：
- 浏览器开发者工具；
- localStorage；

可执行命令：
- 无

可新增状态：
- events
- eventCount

可使用数据存储：
- localStorage

────────────────────
五、约束
────────────────────

禁止修改：
- 宏观进入路径；
- 凝视停留逻辑；
- 镜像回应文案规则；
- 已验收的返回微光行为。

禁止新增：
- 框架；
- 依赖；
- 后端；
- 账号；
- 聊天历史页面。

禁止视觉语义：
- 不出现“已记录”“完成一次反思”“进度 +1”；
- 不让事件保存变成奖励提示。

禁止语言类型：
- 不生成诊断；
- 不新增解释性心理标签。

────────────────────
六、验收标准
────────────────────

A. 运行检验：
- 输入有效文本并完成沉入动作后，eventCount 增加 1；
- localStorage 中存在 events 数组；
- events 数组新增一条数据；
- 刷新页面后 eventCount 不归零。

B. 约束检验：
- 仅修改授权 HTML 文件；
- 无新依赖；
- 无聊天记录 UI；
- 无未授权视觉重做。

C. 语义检验：
- 用户不会感觉自己完成了一次打卡；
- 保存行为不抢占镜像体验；
- 历史仍以对象与空间状态存在，而非列表文字。

D. 回归检验：
- 初始闭环仍可完成；
- 返回宏观的微光仍可出现；
- Escape 仍能退出。

────────────────────
七、验证路径
────────────────────

前置状态：
- 清除 localStorage；
- 页面处于首次进入。

步骤：
1. 点击琥珀对象；
2. 等待进入镜像状态；
3. 输入一句有效文字；
4. 点击回应；
5. 点击“让它沉入其中”；
6. 打开开发期调试区或 localStorage；
7. 刷新页面。

预期：
1. eventCount 从 0 变为 1；
2. localStorage 存在一条事件；
3. 刷新后 eventCount 仍为 1；
4. 无聊天历史列表出现；
5. 原有返回微光仍存在。

失败判定：
- 事件未写入；
- 刷新后丢失；
- 事件重复写入；
- 破坏原有闭环；
- 出现显性打卡或聊天列表。

────────────────────
八、停止条件
────────────────────

保存一条事件并通过刷新验证后立即停止。

禁止继续：
- 不做 candidateSignalCount；
- 不做候选对象；
- 不做命名；
- 不做第二个对象；
- 不做真实 AI。
```

# 6. 任务卡质量检查清单

在进入执行前，产品指挥 Agent 必须检查：

```text
[ ] 目标是否只有一个？
[ ] 是否能由用户观察到？
[ ] 是否明确授权文件？
[ ] 是否明确禁止事项？
[ ] 是否可以由独立验证者重复？
[ ] 是否有明确失败判定？
[ ] 是否有停止条件？
[ ] 是否避免要求执行 Agent “理解整个项目”？
[ ] 是否避免视觉自由发挥？
[ ] 是否避免隐含架构重构？
```

只要有一项为否，任务卡退回重写。

---

# 7. 任务状态流转

```text
Draft
→ Approved
→ In Progress
→ Evidence Submitted
→ Verifying
→ Passed / Failed / Blocked
```

规则：

```text
执行 Agent 无权将任务标记为 Passed；
执行 Agent 只能提交 Evidence Submitted；
Verifier 才能给出 Passed / Failed / Blocked；
Founder 只在 Gate 4 对阶段而非每张卡裁决。
```

---

# 8. 当前使用建议

从下一项实际开发开始，不再使用“把 X 做出来”这种自然语言任务。

必须先由产品指挥 Agent 输出 IU Task Card。  
再交给执行 Agent。  
执行结果交给独立验证 Agent。  
所有失败进入 Failure Ledger。

任务卡的价值不在于让 Agent 更听话，而在于让系统能回答：

> 它到底有没有按正确方式，把正确的事情做出来。
